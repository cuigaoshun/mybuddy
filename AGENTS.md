# AGENTS.md

本文件作用域覆盖整个仓库。

## 项目定位

这是一个陪伴型 Agent 项目。

第一期目标非常明确：

- 只接入飞书。
- 通过飞书 `websocket` 接收消息。
- 通过飞书 API 发送消息。
- 实现 Agent、RAG 和记忆系统的最小闭环。
- 所有存储先用 PostgreSQL 实现，但上层必须依赖抽象接口，不能把业务层写死在 PG 细节上。

除非用户显式要求，不要擅自扩展到企业微信、钉钉、Telegram、Slack 等其他平台。

## 实现边界

后续实现时请严格遵守以下边界：

- `gateway` 层负责外部 IM 平台接入与事件适配。
- `router` 层负责消息路由、会话识别、调用编排。
- `agent` 层负责状态图、决策和回复生成流程。
- `memory` 层负责记忆读写、检索、沉淀和摘要入口。
- `services/im_sender.py` 负责统一发消息能力。
- `event` 层负责内部事件模型和事件传播。
- `workers` 层负责异步任务，不要把后台任务逻辑塞进路由或接入层。

不要把飞书 SDK 调用散落到多个业务模块里。所有平台相关字段都应尽早转换为项目内部统一模型。

## 存储约束

所有存储能力必须先定义抽象，再写 PostgreSQL 实现。

至少要把以下概念隔离为独立仓储接口或等价抽象：

- 会话
- 消息
- 用户记忆
- 事件日志
- 发送记录

禁止把 SQL 或 PG 特有能力直接泄露到 Agent、Router、Gateway 等上层业务模块。

## 代码风格

- 优先小步实现，避免一次性生成大段未经验证的代码。
- 保持现有目录职责稳定，不随意重命名目录。
- 新增代码必须贴合现有模块边界，不要把多层职责混在一个文件里。
- 飞书接入代码优先参考 `tests/lark_on_bot.py` 与 `tests/lark_u_to_bot.py` 的示例方向。
- 文档、命名和注释优先使用清晰直白的中文语义。

## 本仓库特别约束

- 不要擅自编写单元测试。
- 如果需要示例或验证，优先复用已有 `tests` 中的飞书样例作为参考，而不是新增测试体系。

## 文档要求

如果修改架构、一期范围、模块职责或接入方式，需要同步更新 `README.md`，确保 README 仍然可以回答以下问题：

- 这个项目要做什么。
- 第一期做什么，不做什么。
- 飞书如何接入。
- Agent、记忆、RAG、存储分别放在哪一层。
- 为什么虽然当前使用 PG，但仍然要求抽象化。
